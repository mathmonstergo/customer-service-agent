# 导入候选生成任务化与查重 Update 规划

创建时间：2026-05-07 11:12:53 CST

## 修改目标

把现有“逐个同步请求生成候选 FAQ”的批量操作升级为可见的导入生成任务：用户勾选切块后提交任务，前端通过流式状态看到每个切块的排队、处理、跳过、完成或失败状态；后端避免重复提交同一个正在处理或已经生成过候选的切块；生成出的候选 FAQ 自动做查重并标记重复程度。

## 影响范围

- 数据库：新增导入生成任务表、任务切块表，并给候选 FAQ 增加重复检测字段。
- 后端：新增批量生成任务创建接口、SSE 进度接口、任务处理逻辑、候选查重逻辑。
- 前端：批量生成改为创建任务并监听流式进度，切块列表展示状态与跳过原因。
- 测试：覆盖任务幂等、跳过已生成、SSE 事件格式、查重评分和 UI 静态结构。

## 具体方案

- 第一版采用本地单进程同步执行 + SSE 流式输出，不引入 Celery、Redis 或后台 worker。
- 任务创建接口接收 chunk ids，后端创建 job 和 job items；已经 `queued`、`processing` 或 `generated` 的切块直接标记为 `skipped`。
- SSE 接口按 job item 顺序执行生成，推送 `queued`、`processing`、`generated`、`skipped`、`failed`、`done` 事件。
- 查重先用轻量本地算法：
  - 标准化文本 hash 做精确重复。
  - `difflib.SequenceMatcher` 做近似文本重复。
  - 如果候选/正式 FAQ 已有 embedding，再预留 pgvector 语义查重接口；第一版不为了查重强制提前生成 embedding，避免引入大量额外模型调用。
- 候选 FAQ 展示 `duplicate_level`、`duplicate_score`、`duplicate_reason`，保存前给人工审核参考。

## 预期效果

- 用户点击批量生成后能看到每个切块的实时处理过程。
- 重复点击不会让同一切块反复请求 AI。
- 已存在候选的切块默认跳过，必要时后续再加“重新生成”按钮。
- 候选 FAQ 列表能显示重复程度，降低重复保存到标准问答的风险。

## 已确认

- 用户确认可以按“任务化 + 流式状态 + 幂等跳过 + 候选查重”的方向继续。

## 待确认

- 第一版查重先不强制提前生成 embedding；如实际重复误判较多，再接入候选 embedding + pgvector 语义查重。
