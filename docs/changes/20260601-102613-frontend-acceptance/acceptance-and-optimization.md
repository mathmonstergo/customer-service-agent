# 前端验收报告 + 优化方向

- 日期：2026-06-01
- 范围：管理后台 React 前端（文档管理 / FAQ 管理 / 智能问答 三页 + 各自抽屉 + 一次真实 RAG 查询）
- 性质：**验收 / 评审，未改动任何代码**。下方"优化方向"均为待确认项；按 AGENTS.md，任何修复需先与用户确认再实施。
- 方法：Playwright headless（1440×900）逐路由截图 + 控制台/网络监听 + 真实发起一次问答，并对照页面源码（`web/src/pages/**`）逐功能核对行为。

---

## 0. 总体结论（先看这条）

系统**底子很扎实**，不是半成品：

- 三页初始加载 **零** console error / warning / 失败请求。
- 完整 RAG 链路端到端跑通：意图识别 → 向量化 → 混合召回 → 重排 → 流式生成，返回结构良好、有出处的答案，**零运行时报错**。
- 知识治理控件丰富且语义严谨（逐切片重新向量 / 禁用 / 编辑、stale 橙点、假设问题、文档级解析流水线），代码注释质量高。
- "流程详情"抽屉把每一步真实耗时和命中切片都摊开，可解释性是同类内部工具里少见的亮点。

问题集中在**少数显眼的渲染/语义缺陷**和**可解释性的最后一公里**，不涉及架构，多数是低成本高收益的打磨。

## 2. 文档管理

### 2.1 文档列表（`document-list.tsx`）

| 维度 | 评估 |
|---|---|
| 意义 | 文档是知识库主要来源；列表是"解析 + 索引"状态的总览面板。 |
| 用户体验 | 列清晰（名称 / 状态 / 切片·Embedding / 解析器 / 禁用），状态点 + `ready/total` embedding + 过期/失败着色，一眼看出每份文档的索引健康度。 |
| 实际效果 | 渲染正常，状态语义丰富。 |

**问题**
- **[语义 / P1]** "切片 N" 与 "embedding ready/total" 量级不一致且无解释：`total_chunks` 把**假设问题**也算进了知识单元，于是切片数（如 29）和 embedding 分母对不上，用户会困惑。列头叫"切片 / Embedding"但没说明分母 = 切片 + 假设问题。
- **[场景泄漏 / P1]** `{file_type} · {message_count} 条消息`（`document-list.tsx:71`）对**每个**文件都显示。"条消息"只有微信聊天 `.md` 才有意义，PDF/Word/Excel 恒为 `0 条消息`。这正是"通用 UI 不应写死具体场景字段"原则要避免的情况（见记忆 `feedback_generic_ui_not_scenario_specific`）。
- **[语义 / P2]** 所有文档都显示 `待复核`，但部分已完整 embedding。"复核状态"与"可检索状态"的关系不直观，用户不知道待复核是否影响被检索。

### 2.2 文档抽屉 / 切片浏览器（`chunk-browser.tsx`, `document-drawer.tsx`）

| 维度 | 评估 |
|---|---|
| 意义 | 切片是 RAG 检索最小单位；逐切片查看/编辑/重新向量/禁用是知识质量管控的核心，也是与"AI 输出须人工确认"原则对应的人工闸门。 |
| 用户体验 | 极丰富：切片导航 chips（stale 橙点 / 假设问题蓝点 + hover title）、工具栏（section/页码/block_type/`N 问`/embedding 状态 + 重新向量/禁用/编辑）、假设问题折叠块、编辑态"保存后会变 stale"提示、文档级流水线（开始解析/生成 embedding/生成假设问题 + 下载/禁用/删除）。`chunk-browser.tsx:168` 处对 stale 点语义的注释质量很高。 |
| 实际效果 | MinerU **图片资源正常渲染**（chunk #4 = 10 张图，经路径穿越防护的 asset 端点服务，无破图）；切片导航/跳转正常。 |

**问题**
- **[渲染 / P0]** **竖排标题 bug**：chunk #1 "用户使用手册" 每个字独占一行竖排。根因是 MinerU 把标题切成了逐字 block，`SourceBlockPreview` 逐 block 渲染就竖了。这是最显眼的渲染缺陷，直接伤可读性和专业度。
- （同全局）DialogTitle a11y 报错。

## 3. FAQ 管理

### 3.1 列表 / 统计 / 筛选（`faq-list.tsx`, `faq-stats-bar.tsx`）

| 维度 | 评估 |
|---|---|
| 意义 | FAQ 是高置信、人工策划的问答知识，RAG 命中权重高（本次真实查询 5 条来源里 FAQ 占 2 条）。 |
| 用户体验 | 统计条（60 总 / 1 待复核 / 1 未索引 / 0 失败）+ 双维筛选（状态：全部/可用/待复核/草稿/已归档；索引：全部/已索引/未索引/过期/失败）+ 行内信息密集（分类/Q/A 截断/标签/变体点/状态/embedding 点）+ 左侧状态色条。筛选维度完整。 |
| 实际效果 | 渲染正常。 |

**问题**
- **[低价值信息 / P2]** 行首等宽前缀其实是 `faq.id.slice(0, 8)`（`faq-list.tsx:89`）——只是 id 截断（碰巧像 `faq_cand_`/`jszx_qa_` 来源前缀），对用户价值低、易被误读为有意义编码。建议换成真正有用的信息（来源类型/创建时间）或移除。

### 3.2 FAQ 抽屉（`faq-drawer.tsx`）

| 维度 | 评估 |
|---|---|
| 意义 | 单条 FAQ 的人工编辑 + 入库闸门。 |
| 用户体验 | 字段完整：问题* / 相似问法 / 答案* / 分类 / 状态分段 / 置信度分段（高·中·低）/ 标签 + "AI 优化" + 页脚（所有修改已自动保存 / 关闭 / 生成 embedding / 保存修改）。 |

**问题**
- **[交互语义 / P1]** **双重保存模型**：页脚同时出现"所有修改已自动保存"与"保存修改"按钮，自动保存 vs 显式保存语义冲突，用户不知道到底哪个生效、点按钮是否多余。
- **[可解释性 / P2]** 置信度（高/中/低）的来源与作用未说明——不清楚它影响检索权重还是仅人工标注。
- **[风险 / P2]** "AI 优化" 的作用域、是否覆盖原文不明，有改写用户事实的风险（AGENTS.md：AI 输出须经人工确认才入正式库）。建议改写前预览 diff + 显式确认。

## 4. 智能问答（系统价值出口）

| 维度 | 评估 |
|---|---|
| 意义 | 问答是整个系统对外的价值出口，也是 MCP / 微信复用的**同一条** RAG 链路；其余所有功能都为这一步的回答质量服务。 |
| 用户体验 | 空态设计优秀——明确解释"混合召回（向量+关键词）→ 重排 → 流式生成"，并提示右上"流程详情"看每步耗时与命中切片。composer 交互现代：Enter 发送 / Shift+Enter 换行 / Esc 停止 / 自动增高 / 发送-停止按钮切换。 |

**实际效果**（真实查询："测评报告导出一直卡住，应该怎么办？"）
- 流式正常（"停止生成"出现→消失）；回答约 890 字、结构清晰（判断卡住原因 → 按状态尝试 → 重复提交 →【仿真验证】升级信息清单 →【补人工】人工升级）、8 个 markdown 元素、**5 条来源**（FAQ×2 + 文档若干）、**零** error / pageerror / 4xx-5xx。
- **流程详情抽屉是亮点**：6 步均带真实耗时——
  - 输入问题 21ms
  - 意图识别 **9s**（故障排查 / 置信 高 / 含 query 改写）
  - 向量化 1597ms（1024 维）
  - 混合召回 443ms（向量 8 + 关键词 8 → top_k=4 → 召回 5）
  - 命中重排 8ms（最高分 0.66）
  - 生成回答 **13959ms**（输出 826 字）
  - 命中切片 5，带 vector/keyword/FAQ 来源徽章。可解释性对调参/排障极有价值。

**问题**
- **[性能 / P1]** 端到端 ~24s，瓶颈是**意图识别 9s + 生成 14s** 这两段串行。意图识别 9s 偏高（一次 LLM 调用做分类+置信+改写），可与向量化并行、换更快模型或加缓存；生成段受模型限制，但首 token 前等待偏长，体感差。
- **[入口文案 / P2]** "查看 N 条来源"按钮同时承担"打开流程详情"职责，用户可能以为只看来源、不知能看耗时链路（空态虽有提示，但入口文案与实际功能不完全一致）。
- **[模型输出 / P2]** 回答里出现一个疑似乱码 token（"沙包"），是模型输出 artifact 而非系统 bug，但影响观感，可在 prompt / 后处理上收敛。

## 5. 优化方向（按优先级排序的待确认清单）

> 下列均为**建议**，未实施。每项实施前按 AGENTS.md 与用户确认；涉及检索策略/AI 生成逻辑的（P1-④、P2-③）属"较大改动"，需先建 `docs/changes/` 目录走确认流程。

### P0 — 立即可做，低成本高可见（纯前端/配置）
1. **修竖排标题 bug** —— `SourceBlockPreview` 渲染前合并 MinerU 逐字标题 block（或按 block 间距/类型重组）。【§2.2】
2. **消除 DialogTitle a11y 报错** —— 各抽屉补 `DialogTitle`（需隐藏则用 `VisuallyHidden`）。清掉唯一一类控制台报错。【§1】
3. **`<title>` 品牌化** —— 改 `web/index.html`，顺带 favicon。一行。【§1】

### P1 — 语义/体验澄清，中成本
4. **问答延迟优化** —— 意图识别与向量化并行；意图识别换快模型或加缓存；首 token 前给"思考中…"占位强化流式体感。（涉及检索/生成逻辑，需确认）【§4】
5. **文档列表"0 条消息"按类型条件渲染** —— 仅微信 `.md` 显示消息数，其它显示页数/大小/分块数等更相关的指标。【§2.1】
6. **"切片 / Embedding"列加说明** —— tooltip 或副标题点明 embedding 分母 = 切片 + 假设问题（知识单元）。【§2.1】
7. **FAQ 抽屉统一保存模型** —— 二选一：纯自动保存（去掉"保存修改"按钮，保留状态提示）或纯显式保存（去掉自动保存）。【§3.2】

### P2 — 打磨与可解释性
8. **FAQ 行首 id 前缀** 换成有意义信息（来源/时间）或移除。【§3.1】
9. **置信度 / AI 优化 加说明 + 改写预览确认** —— 标注置信度作用；"AI 优化"改写前给 diff 预览并要求确认，守住"人工确认才入库"原则。【§3.2】
10. **澄清"复核状态 vs 可检索状态"** —— 在列表/抽屉用文案或图示说明待复核是否影响被检索。【§2.1】
11. **"查看 N 条来源"入口文案** 调整为同时表达"来源 + 流程耗时"，或拆成两个入口。【§4】
12. **回答乱码 token 收敛** —— prompt 约束或后处理过滤。【§4】

## 6. 明确不动的部分（验收认定为优点，勿改）
- 三页零报错的加载基线、完整且跑通的 RAG 链路。
- "流程详情"抽屉的逐步耗时 + 命中切片可解释性。
- 切片级知识治理控件（重新向量 / 禁用 / 编辑 / stale 语义 / 假设问题）及其代码注释。
- 工具型暗色信息密集布局、空态引导、骨架屏、composer 键盘交互。
